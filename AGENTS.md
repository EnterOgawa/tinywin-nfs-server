# AGENTS.md

## プロジェクト概要
- このプロジェクトは、Java で Windows Pro 上にユーザー空間 NFSv2/NFSv3 サーバーを作り、Windows サービスとして常駐するソフトウェアの開発である。
- 実際のソースコードは `src` フォルダ配下にあり、Java で実装されている。
- QNX 4.25 は重要な検証対象クライアントだが、製品自体は QNX 専用とはしない。
- 製品名は `TinyWinNFS Server` とする。
- Windows サービス ID は `TinyWinNfsServer` とし、旧 ID `OgawaNfsServer` / `QnxNfsServer` は移行用のレガシー名としてのみ扱う。

## 作業前提
- ソース管理は github を利用する。次のアカウントhttps://github.com/EnterOgawaにリポジトリ作成し管理する。
- 作業対象はカレントフォルダ配下に限定し、それ以外の階層は修正しない。
- 環境依存があるため、実行は行わない。
- 確認はソース単位または機能単位で行う。
- eclipseで作業可能なプロジェクトとして作成する。

## 開発環境
- Java は Java 21（JavaSE-21）を使用する。
- 利用する JDK は `C:\develop\tools\eclipse202503\eclipse\jdk-21.0.8+9` とする。
- Eclipse で作業可能な通常の Java プロジェクトとして作成する。
- Eclipse RCP / PDE / OSGi の定義は本プロジェクトには持ち込まない。
- 既存の Eclipse RCP プロジェクトは、コメント形式やプロジェクト運用ルールの参考としてのみ扱う。
- Windows サービス化は、WinSW または Apache Commons Daemon Procrun などのサービスラッパー利用を前提とする。

## 技術スコープ
- 主な検証対象クライアントは QNX 4.25、WSL/Linux NFS クライアントとする。
- 完成目標は NFSv2 / NFSv3 / MOUNT v1-v3 / AUTH_SYS / UDP / TCP の read-write 対応とする。
- 複数の export/mount point を登録できる構成とする。
- read-only は途中段階の検証用状態に限定し、製品仕様としては read-write を必須とする。
- NFSv2/NFSv3 の主要手続きは仕様準拠を目標とし、未実装手続きは明示的に管理する。
- ロック（NLM）、シンボリックリンク、日本語ファイル名、権限マッピングは互換性影響が大きいため、設計を分けて段階的に実装する。

## テスト方針
- 通常確認は、サーバー起動を伴わない単体テストを優先する。
- WSL を利用した NFS mount 確認は、任意の結合テストとして扱う。
- WSL 結合テストでは、管理者権限が必要な低番ポートを避け、`11111` / `12049` / `12048` の高番ポートを使用する。
- WSL 結合テスト、NFSサーバー起動、サービス起動は環境依存のため、ユーザーから明示的な実行指示がある場合のみ実施する。

## 作業ルール
1. 作業場所は `work` フォルダ以下とする。
2. スクリプトやアウトプットファイルは全てフォルダを分けて管理し、`work` フォルダ直下は常にクリーンに保つ。
3. ファイル出力が必要な解析結果は、`work/analysis` 配下に出力フォルダを作成して格納する。
4. ソース変更の提示は、必ず `work` フォルダ配下に `.md` 形式ファイルとして出力する。
5. 今後の修正提案ファイルは `work/proposals` 配下に作成し、ファイル名は `YYYYMMDD_HHMMSS_***.md` 形式とする。
6. 作業用の一時ファイルは `work/tmp` を使用し、永続保管は前提としない。
7. コメントはsample/AplMain.javaの形式に合わせること。また、Javadoc 準拠とし、警告を残さないこと。

## 修正提案の提示ルール
- 変更対象のファイル名、メソッド名、行番号を明示すること。
- 提示するコードは変更箇所のみとし、各行先頭を `-` と `+` で示すこと。
- 修正提案の `.md` ファイルは `work/proposals` 配下に作成すること。

## コーディング上の注意
1. `if` 文では条件式コメントを必ず付与すること。
2. `for` 文の前には処理意図が分かるコメントを必ず付与すること。
3. `if{}` や `for{}` などの閉じかっこの下には、必ず 1 行のスペースを空けてから次のソースを記述すること。
4. ただし `if{} else{}` の場合は、`if{}` の閉じかっこの後ろの行間は不要とする。
5. `if{}` の直後が `//else` コメントの場合も、`if{}` の閉じかっこの後ろの行間は不要とする。
6. `if{} else{}` の間にコメント行が入る場合も、`if{}` の閉じかっこの後ろの行間は不要とする。
7. `try{} catch{}` についても `if{} else{}` と同等に扱い、`try{}` と `catch{}` の間はコメント行を含めて行間を空けなくてもよい。
8. `try{} catch{}` の場合、`try{}` の閉じかっこの後ろには 1 行のスペースは不要とする。
9. `switch` 文では、`case` の上にある `break` の直下に 1 行の空白を入れること。
